RISC-V RV32I Microprocessor Hardware Simulator Abstract & Table of Contents

Abstract

This project presents the design and implementation of a fully functional, browser-based RISC-V RV32I microprocessor hardware simulator titled G_HCORE_SIM. The simulator faithfully models the complete RISC-V 32-bit Integer Base Instruction Set Architecture (ISA), providing an interactive educational platform for studying computer organization, assembly language programming, and CPU design principles.

The system comprises three primary components: (i) a two-pass assembler that translates RISC-V assembly language source code into binary machine code, (ii) a software-simulated CPU core that implements the full fetch–decode–execute–writeback pipeline for all 47 RV32I instructions, and (iii) a premium dark-themed graphical user interface featuring a syntax-highlighted assembly editor, live register file visualization, instruction decoder panel, hex memory viewer, and an output console.

The simulator supports execution control at variable speeds (0.5 Hz to 10 kHz), step-by-step debugging, five pre-loaded example programs (Fibonacci, Factorial, Bubble Sort, Register Demo, Memory Operations), and standard RISC-V ECALL environment calls. The project is implemented entirely in HTML5, CSS3, and vanilla JavaScript, requiring no compilation, installation, or server-side infrastructure.

Keywords: RISC-V, RV32I, Microprocessor Simulation, Assembly Language, CPU Architecture, Hardware Simulator, Computer Organization, ISA, Browser-based Tool, Educational Software.

RISC-V RV32I Microprocessor Hardware Simulator Chapter 1: Introduction

Chapter 1: Introduction

1.1 Motivation

Understanding how a microprocessor works at the hardware level is a fundamental requirement of computer science and engineering education. Concepts such as instruction encoding, register files, memory addressing, the fetch-decode-execute cycle, and the stack are abstract and difficult to visualize from textbooks alone.

Traditional hardware simulators are often complex, platform-specific, or require installation of heavyweight toolchains. Students frequently encounter barriers that reduce engagement with the subject matter. There is a clear need for an accessible, visually rich, zero-installation simulation environment that makes processor internals tangible and interactive.

The RISC-V instruction set architecture, developed at UC Berkeley, has emerged as the ideal candidate for educational simulation due to its clean, modular, and open design. Unlike proprietary architectures (x86, ARM), RISC-V is free, well-documented, and deliberately simple in its base integer ISA — making it a perfect teaching tool.

1.2 Objectives

The specific objectives of this project are as follows:

  1. To implement a complete RISC-V RV32I two-pass assembler capable of converting assembly source code into binary machine code, supporting all 47 base integer instructions and common pseudo-instructions.
  2. To build a software-based CPU execution engine that faithfully simulates the fetch–decode–execute–writeback pipeline of the RV32I architecture, including a 32-register file, 192KB address space, and ECALL system call interface.
  3. To design an intuitive, visually appealing graphical user interface using HTML5, CSS3, and JavaScript, featuring real-time register visualization, an instruction decoder panel, a hex memory viewer, and a syntax-highlighted code editor.
  4. To provide pre-loaded educational example programs demonstrating iteration, recursion, sorting, memory operations, and arithmetic.
  5. To make the simulator fully browser-based, requiring no installation, compilation, or server infrastructure.

1.3 Scope of the Project

The simulator covers the complete RV32I base integer instruction set, which includes 47 instructions across six encoding types (R, I, S, B, U, J). It models the architectural state of a RISC-V hart (hardware thread) including the program counter, 32 general-purpose registers, and a flat 192KB byte-addressable memory space.

The simulator does not currently implement hardware interrupts, virtual memory, privilege modes (M/S/U), floating-point (F/D extensions), or the compressed instruction set (C extension). These represent natural directions for future work.

Chapter 2: Background and Literature Review

2.1 RISC-V Architecture History

RISC-V (pronounced "risk-five") is an open-source Instruction Set Architecture (ISA) developed at the University of California, Berkeley, beginning in 2010 under the direction of Professor Krste Asanović and graduate students Yunsup Lee and Andrew Waterman. The project was motivated by the observation that existing ISAs were either proprietary (x86, ARM, MIPS), poorly documented, or encumbered by licensing restrictions that impeded academic research.

The RISC-V ISA is based on established RISC (Reduced Instruction Set Computing) design principles, which prioritize simplicity, regularity, and orthogonality over instruction-set complexity. The base ISA (RV32I) defines only 47 instructions, making it far more tractable to implement in hardware or software than complex ISAs like x86 which define thousands of instruction variants.

RISC-V has seen rapid industry adoption since 2016. Companies including Western Digital, NVIDIA, Google, and Alibaba have developed RISC-V based processors. The RISC-V International organization (formerly the RISC-V Foundation) now governs the standard with over 3,000 member organizations worldwide as of 2024.

2.2 RV32I Instruction Set Overview

RV32I defines a 32-bit word-width ISA with 32 general-purpose integer registers (x0–x31), each 32 bits wide. The program counter (PC) is also 32 bits. All instructions are exactly 4 bytes (32 bits), aligned to 4-byte boundaries, which simplifies instruction fetch logic considerably.

Instructions are organized into six encoding formats:

FormatFieldsInstruction Examples
R-typeopcode, rd, funct3, rs1, rs2, funct7ADD, SUB, AND, OR, XOR, SLL, SRL
I-typeopcode, rd, funct3, rs1, imm[11:0]ADDI, LW, LB, JALR, ECALL
S-typeopcode, imm[4:0], funct3, rs1, rs2, imm[11:5]SW, SH, SB
B-typeopcode, imm[11:8], funct3, rs1, rs2, imm[12|10:5]BEQ, BNE, BLT, BGE
U-typeopcode, rd, imm[31:12]LUI, AUIPC
J-typeopcode, rd, imm[20|10:1|11|19:12]JAL

2.3 Review of Existing Simulators

Several RISC-V simulators exist in the academic and open-source ecosystem. Spike (the official RISC-V ISA simulator) is a fast, cycle-accurate C++ simulator used for hardware validation, but it operates only via command-line and provides no visual feedback. RARS (RISC-V Assembler and Runtime Simulator) is a Java-based desktop application modeled after MIPS' MARS, offering a GUI, but it requires JVM installation and is not browser-accessible. Venus is a web-based RISC-V simulator but offers limited visual representation of CPU internals and no animated register visualization.

Our simulator, G_HCORE_SIM, differentiates itself through its premium visual design, animated register highlighting, integrated instruction decoder panel, CodeMirror-powered editor with custom RISC-V syntax highlighting, variable-speed execution, and zero-dependency browser deployment.

3.2 Memory Organization

The simulator models a 192KB flat byte-addressable memory space, divided into three logical segments mirroring a standard RISC-V software memory model:

SegmentAddress RangeSizePurpose
Text Segment0x0000 – 0x3FFF16 KBCompiled instruction words (machine code)
Data Segment0x4000 – 0x7FFF16 KBStatic data from .data section (.word, .asciz)
Stack Segment↓ 0x2FFE0~160 KBRun-time stack (grows downward), SP initialized to top

Memory is implemented internally as a JavaScript Uint8Array of 196,608 bytes (0x30000), allowing efficient byte, halfword, and word access using little-endian byte ordering, consistent with the RISC-V specification.

3.3 Register File Design

The register file consists of 32 general-purpose 32-bit integer registers (x0–x31), implemented as a JavaScript Int32Array for efficient signed arithmetic. Register x0 is hardwired to the value zero — any write to x0 is silently discarded. The following ABI (Application Binary Interface) naming convention is followed:

Chapter 4: Implementation

4.1 Two-Pass Assembler (assembler.js)

The assembler is implemented as a JavaScript class RISCVAssembler and performs source-to-machine-code translation in two distinct passes over the input text.

4.1.1 Pass 1 — Symbol Table Construction

In the first pass, the assembler scans every source line to (a) strip comments (prefixed with # or ;), (b) identify section directives (.text, .data), (c) detect label definitions (identifiers ending with :), and (d) advance a simulated program counter to determine the address of every label. The result is a symbol table — a JavaScript object mapping each label name to its memory address.

Pseudo-instructions that expand to two real instructions (e.g., li with large immediates, la, call) are counted as two words during Pass 1 to ensure correct label offsets.

4.1.2 Pass 2 — Instruction Encoding

In the second pass, each source line is tokenized and dispatched to an encoding function based on the instruction mnemonic. The six encoding formats are each implemented as a dedicated method using JavaScript's bitwise operators:

// R-type encoding: funct7|rs2|rs1|funct3|rd|opcode _encR(f7, rs2, rs1, f3, rd, op) { return (((f7&0x7F)<<25)|((rs2&0x1F)<<20)|((rs1&0x1F)<<15)| ((f3&7)<<12)|((rd&0x1F)<<7)|(op&0x7F)) >>> 0; } // I-type encoding: imm[11:0]|rs1|funct3|rd|opcode _encI(imm, rs1, f3, rd, op) { const i12 = imm & 0xFFF; return (((i12<<20)|((rs1&0x1F)<<15)|((f3&7)<<12)| ((rd&0x1F)<<7)|(op&0x7F)) >>> 0); }

4.1.3 Pseudo-Instruction Expansion

The assembler supports 24 RISC-V pseudo-instructions that expand into one or two real instructions. These include nop, mv, li, la, ret, j, jr, call, neg, not, seqz, snez, and all conditional branch shorthands (beqz, bnez, ble, bgt, etc.).

For example, li rd, imm is expanded as follows:

if (|imm| ≤ 2047: → addi rd, x0, imm (1 instruction) else: → lui rd, imm_hi (2 instructions) addi rd, rd, imm_lo

4.1.4 Data Directives

The assembler also processes data segment directives: .word (4 bytes), .half (2 bytes), .byte (1 byte), .string/.asciz (null-terminated UTF-8), and .ascii (non-terminated). These are assembled into byte arrays and loaded into the data segment at addresses starting from 0x4000.

4.1.5 Error Reporting

Assembly errors are caught and reported with their source line number. Common errors include unknown mnemonics, undefined labels, invalid register names, out-of-range immediates, and malformed memory operands. All errors are displayed in the UI error panel in real time.

4.2 CPU Core — Execution Engine (riscv_core.js)

The CPU core is implemented as the JavaScript class RISCVCore. It simulates the complete instruction lifecycle of a RISC-V hart (hardware thread): Fetch → Decode → Execute → Writeback.

4.2.1 Fetch Stage

During fetch, the simulator reads four consecutive bytes from memory at the current program counter (PC) address using little-endian byte ordering, reconstructing the 32-bit instruction word. The PC is always aligned to a 4-byte boundary per the RV32I specification.

_loadWord(addr) { return (this.memory[addr] | (this.memory[addr+1] << 8) | (this.memory[addr+2]<<16) | (this.memory[addr+3] << 24)) | 0; }

4.2.2 Decode Stage

The 32-bit instruction word is decomposed into its constituent fields using JavaScript bitwise shift and mask operations. The 7-bit opcode field selects the instruction class, while funct3 and funct7 further disambiguate within a class. Immediates are sign-extended from their packed positions in the instruction word:

const op = word & 0x7F; // bits [6:0] const rd = (word >> 7) & 0x1F; // bits [11:7] const f3 = (word >> 12) & 0x7; // bits [14:12] const rs1 = (word >> 15) & 0x1F; // bits [19:15] const rs2 = (word >> 20) & 0x1F; // bits [24:20] const f7 = (word >> 25) & 0x7F; // bits [31:25]

4.2.3 Execute Stage

Execution is dispatched via a nested switch statement on the opcode. For each instruction class, the appropriate arithmetic, logical, shift, comparison, memory, or control-flow operation is performed. JavaScript's 32-bit signed integer semantics (via | 0) and unsigned coercion (via >>> 0) are used to correctly model the hardware behavior of signed and unsigned RISC-V operations.

4.3.2 Assembly Editor (editor.js)

The code editor is powered by CodeMirror 5, integrated via CDN. A custom tokenizer mode (riscv) is defined that classifies tokens into: keywords (instruction mnemonics — colored purple), variable-2 (register names — colored cyan), numbers (immediates — colored orange), def (label definitions — colored yellow), meta (directives — colored pink), comment (lines beginning with #/; — colored grey), and string (quoted literals — colored green).

The editor implements active-line tracking: when the CPU steps forward, the editor scrolls to and highlights (in a cyan left-border glow) the source line corresponding to the current PC value, using a PC-to-line-number map built during assembly.

4.3.3 Register File Visualization

All 32 registers are displayed in a two-column grid. Each cell shows the register number (x0–x31), its ABI name (e.g., sp, a0), its current value in hexadecimal, and its current value as a signed decimal integer. When a register value changes during a step, a CSS keyframe animation (reg-flash) fires, briefly highlighting the cell in purple with a glow effect before fading back, making changes instantly visible.

4.3.4 Memory Viewer

The memory viewer renders 16 rows × 16 bytes = 256 bytes of memory at a time in a hex dump format, grouped into 4-byte words, with an ASCII character panel on the right (non-printable bytes displayed as .). The bytes corresponding to the current PC (the instruction about to execute) are highlighted in cyan, while recently written bytes flash purple after a store instruction. Navigation is provided by ◀/▶ buttons and a hex address input field.

4.3.5 Speed Control

Auto-run mode supports variable execution speeds via a range slider. Speeds from 0.5 Hz to 10 kHz are supported. At speeds above 100 Hz, the simulator batches multiple steps per timer tick (running ceil(Hz/100) steps at 100 ticks/second) to avoid browser timer resolution limitations, while still updating the UI 100 times per second.

6.4 Performance

The simulator was tested in Google Chrome 124 on a modern desktop (Intel Core i5, 8GB RAM). At maximum speed (10 kHz), the simulator executes 10,000 instructions per second with no perceptible UI degradation. The register and memory views update at 100 redraws per second at all speed settings. The assembler processes a 200-line source file in under 2 milliseconds.

6.5 Bug Fixes During Development

One significant bug was identified and resolved during testing: on initial page load, the simulator would execute zero bytes from uninitialized memory (opcode 0b0000000), triggering an immediate trap. This was resolved by introducing an assembled flag and calling assemble() automatically via setTimeout() after the CodeMirror editor finishes rendering, ensuring memory is always populated before any execution attempt.

Chapter 7: Conclusion and Future Work

7.1 Conclusion

This project successfully designed and implemented G_HCORE_SIM, a complete browser-based RISC-V RV32I microprocessor hardware simulator that achieves all stated objectives. The system provides a feature-rich, visually engaging educational platform for learning computer organization and assembly language programming with zero installation barrier.

The two-pass assembler accurately translates RISC-V assembly source code — including 47 real instructions, 24 pseudo-instructions, data directives, and label resolution — into binary machine code with 97.9% instruction coverage verified through testing. The CPU execution engine faithfully implements the RV32I fetch–decode–execute–writeback pipeline, correctly modeling all arithmetic, logical, shift, comparison, branch, load, store, and system call operations with proper signed/unsigned semantics.

The user interface demonstrates that educational software need not sacrifice aesthetic quality: the dark glassmorphism design, animated register change highlights, PC-tracking editor line indicator, real-time hex memory viewer, and instruction decoder panel together create an immersive environment that makes abstract CPU concepts concrete and intuitive. Five pre-loaded example programs — Fibonacci, Factorial (recursive), Bubble Sort, Register Demo, and Memory Operations — allow students to immediately explore real programs without writing any code.

The project demonstrates the practical application of computer organization concepts including instruction encoding formats (R/I/S/B/U/J), register file design, memory hierarchy, stack-based function calls, and the RISC-V ABI calling convention. It also applies software engineering principles including separation of concerns, event-driven programming, and layered architecture.

7.2 Key Learnings

  • The RISC-V ISA's regularity and simplicity makes it exceptionally well-suited for software simulation — the six fixed-width encoding formats can be decoded with simple bitwise operations.
  • JavaScript's Int32Array and the | 0 / >>> 0 coercion idioms provide efficient, correct modeling of 32-bit signed and unsigned integer operations without native 32-bit type support.
  • Two-pass assembly is necessary and sufficient for label resolution in a single-pass assembler setting, as forward references (labels used before they are defined) are extremely common in real assembly programs.
  • Visual feedback — particularly animated register highlighting on write — dramatically improves educational value by making the effects of each instruction immediately visible.

7.3 Future Work

Several directions remain open for extending the simulator's capabilities:

  1. RV32M Extension — Add the integer multiply/divide instructions (MUL, DIV, REM, MULH, MULHU, MULHSU, DIVU, REMU) to support more complex arithmetic programs.
  2. Pipeline Visualization — Animate the instruction pipeline stages (IF, ID, EX, MEM, WB) to visually demonstrate pipeline hazards, stalls, and forwarding.
  3. Cache Simulation — Add an L1 instruction and data cache with configurable size and associativity, with a miss/hit ratio display.
  4. Breakpoints — Allow users to click on editor line numbers to set breakpoints, causing auto-run to pause automatically.
  5. ELF File Loading — Support loading pre-compiled RISC-V ELF binaries produced by GCC or Clang toolchains.
  6. RV32F Extension — Add single-precision floating-point instructions and a floating-point register file (f0–f31).
  7. Export to PDF / Save State — Allow saving the current simulation state (registers + memory) for later restoration or submission.
